    Assertion

    总结
    断言是一种调试方式，断言失败会抛出Assertion
    只能在开发和测试阶段启用断言
    对可恢复的错误不能使用断言，应抛出异常
    断言很少被使用，更好的方法是编写测试单元

    断言是一种程序调试方式

    • 使用assert关键字
    • 断言条件预期为true
    • 如果断言失败，抛出AssertionError

    断言的特点：

    • 断言失败时会抛出AssertionError,导致程序结束退出
    • 不能用于可恢复的程序错误
    • 只应该用于开发和测试阶段

    对于可恢复的程序错误：

    • 不应该使用断言
    • 应该抛出异常并在上层捕获

    启用断言

    • JVM默认是关闭断言指令的
      • 给JVM传递-ea参数启用断言
        选择Class 右键，run as 然后进入 run configurations，进入Arguments修改 VM
        arguments参数，输入完整类名：-ea:包名+类名
      • 可以指定特定的类启用断言
      -ea:com.xxx.Main
      • 可以指定特定的包启用断言
      **-ea:com.xxx…

    Logging

    总结
    日志可以定义格式，重定向到文件
    日志可以存档，便于追踪问题
    日志记录可以按级别分类，便于打开或关闭某些级别
    可以根据配置文件调整日志，无需修改代码
    JDK提供了 Logging:java.util.logging

    什么是日志？

    • 为了取代System.out.println();println不具备下述能力。
    • 设置输出样式
    • 设置级别，进制某些级别输出
    • 可以被重定向到文件
    • 可以按包名控制日志级别

    JDK Logging

    • JDK内置了Logging，在java.util.logging
    JDK Logging定义了7个日志级别
    SEVERE
    VARNING
    INFO 默认级别
    CONFIG
    FINE
    FINER
    FINEST

    JDK Logging的局限：

    • JVM 启动就需要读取配置文件并初始化完成
    • JVM 启动后无法修改配置
    • 需要在JVM启动时传递参数(-Djava.util.loggin.config.file=config-file-name)

    Commons Logging

    • Commons Logging是使用最方法的日志模块
    • Commons Logging的API非常简单
    • Commons Logging可以自动使用其他日志模块

    Commons Logging 是Apache创建的日志模块：

    • 可以挂载不同的日志系统
    • 可以通过配置文件指定挂接的日志系统
    • 自动搜索并使用Log4j
    • 使用JDK Logging
    日志级别
    FATAL 非常严重
    ERROR 普通错误
    WARNING
    INFO 默认
    DEBUG
    TRACE 更底层的调试信息

    Log4j

    前面介绍了Commons Logging，可以作为“日志接口”来使用。而真正的“日志实现”可以使用Log4j。

    Log4j是一种非常流行的日志框架，最新版本是2.x。

    Log4j是一个组件化设计的日志系统，它的架构大致如下：

    log.info(“User signed in.”);
    │
    │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
    ├──>│ Appender │───>│ Filter │───>│ Layout │───>│ Console │
    │ └──────────┘ └──────────┘ └──────────┘ └──────────┘
    │
    │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
    ├──>│ Appender │───>│ Filter │───>│ Layout │───>│ File │
    │ └──────────┘ └──────────┘ └──────────┘ └──────────┘
    │
    │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
    └──>│ Appender │───>│ Filter │───>│ Layout │───>│ Socket │
    └──────────┘ └──────────┘ └──────────┘ └──────────┘
    当我们使用Log4j输出一条日志时，Log4j自动通过不同的Appender把同一条日志输出到不同的目的地。例如：

    console：输出到屏幕；
    file：输出到文件；
    socket：通过网络输出到远程计算机；
    jdbc：输出到数据库
    在输出日志的过程中，通过Filter来过滤哪些log需要被输出，哪些log不需要被输出。例如，仅输出ERROR级别的日志。

    最后，通过Layout来格式化日志信息，例如，自动添加日期、时间、方法名称等信息。

    上述结构虽然复杂，但我们在实际使用的时候，并不需要关心Log4j的API，而是通过配置文件来配置它。

    • Commons Logging 如果在classpath中发现了log4j,就会使用log4j

    最佳实践

    • 始终使用Commons Logging接口来写入日志
    • 开发阶段无需引入Log4j
    • 使用Log4j只需要把正确的配置文件和相关的jar包放入classpath
    • 使用配置文件可灵活修改日志，无需修改代码

    总结

    通过Commons Logging实现日志，不需要修改代码就可以使用Log4j
    使用Log4j只需要把log4j2.xml和相关的jar放入classpath
    如果要更换Log4j,只需要移除log4j2.xml和相关的jar
    只有扩展Log4j,才需要引用Log4j的接口
